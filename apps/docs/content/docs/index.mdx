Welcome to the **Prood** documentation. This monorepo ships a complete white-label commerce platform: a customer storefront, merchant dashboard, standalone checkout app, REST API, and shared packages for catalog, payments, and storage.

## What is Prood?

Prood is a **commerce-agnostic**, **multi-tenant** toolkit. Each merchant runs as a Better Auth **organization**; all commerce data is isolated by Postgres **row-level security (RLS)**. You can swap payment providers, storage backends, and (in the future) commerce adapters without rewriting your apps.

## Core packages

| Package | Role |
| --- | --- |
| `@prood/types` | Unified domain types and provider interfaces |
| `@prood/platform` | Built-in commerce engine (Neon Postgres + Drizzle + RLS) |
| `@prood/commerce` | Server-only data layer wrapping the adapter and providers |
| `@prood/checkout` | Framework-agnostic checkout state machine |
| `@prood/checkout-host` | Next.js session host (Upstash Redis) |
| `@prood/api-client` | Typed OpenAPI fetch client |
| `@prood/ui` | shadcn/Radix + 33+ commerce UI components |

## How data flows

```
Storefront / Dashboard  ──HTTP──►  apps/api  ──►  @prood/commerce  ──►  @prood/platform (Neon)
       │                                                                  │
       └── CHECKOUT_URL ──►  apps/checkout  ──webhooks──►  apps/api
```

The **storefront** and **dashboard** never talk to the database for commerce data directly. They call the **Commerce API** via `@prood/api-client`. Only `apps/api` (and dashboard integration helpers) invoke `@prood/commerce`, which scopes every query to the active tenant with `withTenant()`.

## Key concepts

### Multi-tenancy

Each merchant store is a Better Auth organization. The storefront resolves the tenant from the request **Host** header (custom domain or `{slug}.{platform}` subdomain). The dashboard uses the session's **active organization**. See [Multi-tenant platform](/docs/architecture/multi-tenant).

### API-centric architecture

Unlike the upstream Nuxt reference (where apps imported `@prood/core` directly), this Next.js stack routes all commerce operations through `apps/api`. Benefits:

- Single auth and tenant resolution layer
- OpenAPI contract for humans, agents, and MCP tools
- Storefront stays thin — BFF routes only for cart cookies and client-side search

### Checkout separation

Order placement happens on the storefront (via the API). Payment UI lives in `apps/checkout`, with sessions persisted in **Upstash Redis**. After payment, webhooks reconcile order status through the API.
